# Blueprint Catalog Seed

Exported snapshot of the Supabase `blueprints` table (`scope='catalog'`) — the
insurance copy and the editable template library. Generated by
[`scripts/export-catalog.mjs`](../scripts/export-catalog.mjs).

## Why this exists

The catalog is being rebuilt from scratch with real-business agents (the
three-layer architecture's "catalog cut"). Before the fictional content is
cleared, it's archived here so it can be re-seeded later, and so new blueprints
can be authored against a known-good template.

## The wipe boundary

| File | `kind` | Count | Disposition |
|---|---|---|---|
| `catalog/capability-agents.json` | `capability` | 20 | **KEEP** — MCP-wired umbrella agents, load-bearing |
| `catalog/character-agents.json` | `character` | 489 | Disposable — fictional persona agents |
| `catalog/spaceships.json` | `spaceship` | 24 | Disposable — fictional ships |

`capability` agents wrap real MCP integrations (HubSpot, GitHub, Slack…). They
are the functional core and must survive the wipe. `character` and `spaceship`
blueprints are the fictional layer being replaced — note that spaceships pin
specific character agents by id via `config.crew_overrides`, so the two
disposable files are coupled and get cleared together.

## Blueprint shape

Every blueprint row has these columns:

| Column | Type | Notes |
|---|---|---|
| `id` | string | stable key — `bp-agent-github`, `ship-50`, `the-matrix` |
| `serial_key` | string | display serial — `CR-GTHB-AGNT-0001-NICE` |
| `type` | enum | `agent` or `spaceship` |
| `kind` | enum | `capability` / `character` / `spaceship` |
| `name`, `description`, `flavor` | string | display text |
| `category` | string | Engineering, Marketing, Automation, … |
| `rarity` | enum | Common / Rare / Epic / Legendary / Mythic |
| `tags` | string[] | search/filter tags |
| `config`, `stats`, `metadata` | object | shape varies by `kind` — see below |
| `is_public` | bool | catalog rows are `true` |

Runtime/derived columns (`creator_id`, `rating_avg`, `activation_count`,
`search_vector`, `created_at`, `updated_at`) are **not** exported — Supabase
sets them on insert, so the JSON round-trips cleanly.

### `config` by kind

**capability agent** — the function, no persona:
- `tools` — array of real MCP tool ids (`search_code`, `slack_read_thread`, …)
- `system_prompt` — operational prompt: how to use the tools effectively
- `role`, `role_type` — function label (`Engineering` / `engineering`)
- `llm_engine`, `temperature`, `maxSteps`, `memory` — execution params

**character agent** — the persona, no function of its own:
- `system_prompt` — the *persona* prompt (voice, character, tone)
- `capability_id` — optional; points at a capability blueprint's `id`. When
  set, the character is "wired" — at execution time `AgentExecutor` resolves
  tools from the capability, not the character. 16 of 489 are wired.
- `ship_id` — which spaceship this character belongs to
- `tools` — `[]` for stub characters; mirrors the capability for wired ones
- `role`, `role_type`, `llm_engine`, `temperature`, `maxSteps`, `memory`

**spaceship** — the workflow + crew composition:
- `crew_roles` — array of role tokens, length = slot count. `CrewMatcher`
  fills slots by matching these against agent `config.role` / `category`.
- `crew_overrides` — `{ "<slotIndex>": "<agentBlueprintId>" }` — pinned crew,
  bypasses matching. A ship uses one or the other (18 use `crew_roles`,
  6 use `crew_overrides`).
- optional: `flow`, `class`, `crew_size`, `auto_theme`, `integrations`

### `stats` by kind

- **agent**: `acc`, `cap`, `pwr`, `spd` — display strings (`"92%"`, `"3.5s"`)
- **spaceship**: `crew`, `slots`, optionally `tier`, `power`, `speed`, `defense`

### `metadata` by kind

- **agent**: `caps` (capability bullet list), `art`, `flavor`, `card_num`,
  `agentType`. Capability agents also carry `tools_required` (e.g.
  `["github"]` — which MCP must be connected).
- **spaceship**: `caps`, `crew` (named crew array — each `{ id, type, label,
  config.agentRole, rarity }`), `card_num`, `recommended_class`, optionally
  `tier` / `author` / `version` / `skin_pack`

## Authoring a new blueprint

1. Copy the closest existing entry from the matching file as a starting point.
2. Give it a unique `id` and `serial_key`.
3. Fill `config` / `stats` / `metadata` per the kind shape above.
4. For a `capability` agent: list the real MCP `tools`, write the operational
   `system_prompt`, set `metadata.tools_required` to the MCP name.
5. For a `character`: write the persona `system_prompt`; set `capability_id`
   to wire it to a capability, or leave it unset for a persona-only stub.
6. For a `spaceship`: set `stats.slots`, then either `config.crew_roles`
   (length = slots) or `config.crew_overrides`.

## Re-seeding

These files are not yet auto-applied — re-seeding tooling is Phase 2 of the
catalog rebuild. The intended path, mirroring `scripts/seed-crew-roles.mjs`, is
to emit a SQL migration that upserts the JSON into `blueprints`
(`INSERT … ON CONFLICT (id) DO UPDATE`). The old
`supabase/migrations/20260323000003_seed_full_catalog.sql` is the *pre-reset*
924-row catalog and is **not** a rebuild path.

## Refreshing this export

```
node scripts/export-catalog.mjs              # scope=catalog (default)
node scripts/export-catalog.mjs --scope all  # also community + system
```

Read-only — the script never writes to Supabase.
